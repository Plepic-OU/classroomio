# BDD Tests with Gherkin & Playwright — Design Document

**Date:** 2026-03-13
**Status:** Draft
**Scope:** Initial setup — login and course creation flows only

---

## Overview

Set up BDD testing for ClassroomIO using `playwright-bdd` (Gherkin integration on top of Playwright's native test runner). The initial scope covers two flows: login and course creation. Tests run against the local devcontainer environment and the Playwright HTML report is accessible from the host machine on port 9323.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| BDD framework | `playwright-bdd` | Keeps Playwright's native runner, parallel execution, traces, and HTML report. Lighter than `@cucumber/cucumber`. |
| Package location | `packages/e2e/` | Standalone workspace package. Scales across apps, consistent with monorepo pattern. |
| Step definition org | Per feature file | Simple with only 2 flows. Refactor to domain grouping later if reuse grows. |
| Test data strategy | Supabase seed script | Reliable, flows run independently. No dependency on signup working. |
| Report port | 9323 | Playwright's default. Added to devcontainer port forwarding. |

## Package Structure

```
packages/e2e/
├── package.json
├── playwright.config.ts
├── tsconfig.json
├── seed/
│   └── test-users.ts           # Supabase seed script for test data
├── fixtures/
│   └── auth.fixture.ts         # Playwright fixture: authenticated session
├── features/
│   ├── login.feature            # Gherkin: login flow
│   └── course-creation.feature  # Gherkin: course creation flow
├── steps/
│   ├── login.steps.ts           # Step definitions for login
│   └── course-creation.steps.ts # Step definitions for course creation
├── .features-gen/               # Git-ignored, generated Playwright specs
└── .auth/                       # Git-ignored, stores session storageState
```

### Dependencies

- `@playwright/test` — test runner
- `playwright-bdd` — Gherkin-to-Playwright integration
- `@supabase/supabase-js` — seed script (connects to local Supabase)

### Scripts (`packages/e2e/package.json`)

| Script | Command | Description |
|--------|---------|-------------|
| `generate` | `npx bddgen` | Generate Playwright specs from `.feature` files |
| `test` | `npx bddgen && npx playwright test` | Generate + run all BDD tests |
| `test:ui` | `npx bddgen && npx playwright test --ui` | Interactive Playwright UI mode |
| `report` | `npx playwright show-report --port 9323` | Serve HTML report on port 9323 |
| `seed` | `npx tsx seed/test-users.ts` | Insert test data into local Supabase |

## Playwright & playwright-bdd Configuration

### `playwright.config.ts`

- **Base URL:** `http://localhost:5173` (dashboard dev server)
- **Browser:** Chromium only
- **Test directory:** `.features-gen/` (generated by `bddgen`)
- **Reporter:** HTML reporter (port 9323)
- **Trace:** `on-first-retry`
- **`defineBddConfig()`** — points `features` to `./features/`, `steps` to `./steps/`

### Test Projects

Two Playwright projects (not monorepo packages):

1. **`setup`** — Runs the seed script, logs in via the UI, saves `storageState` to `.auth/user.json`. No `.feature` files.
2. **`tests`** — Depends on `setup`. Runs all `.feature` flows with the saved auth state.

This means login feature tests exercise the actual login UI, while course creation tests skip login by reusing the stored session.

## Feature Files

### `features/login.feature`

```gherkin
Feature: Login

  Scenario: Successful login with valid credentials
    Given I am on the login page
    When I enter the email "testuser@classroomio.test"
    And I enter the password "TestPass123!"
    And I click the login button
    Then I should be redirected to the courses page

  Scenario: Login fails with invalid password
    Given I am on the login page
    When I enter the email "testuser@classroomio.test"
    And I enter the password "wrongpassword"
    And I click the login button
    Then I should see an error message
```

### `features/course-creation.feature`

```gherkin
Feature: Course Creation

  Scenario: Create a live class course
    Given I am logged in
    And I am on the courses page
    When I click the new course button
    And I select "Live Class" as the course type
    And I enter "BDD Test Course" as the course title
    And I enter "A test course description" as the description
    And I click the create button
    Then I should see "BDD Test Course" in the course list

  Scenario: Create a self-paced course
    Given I am logged in
    And I am on the courses page
    When I click the new course button
    And I select "Self Paced" as the course type
    And I enter "Self Paced Test" as the course title
    And I enter "Self paced description" as the description
    And I click the create button
    Then I should see "Self Paced Test" in the course list
```

## Seed Script & Auth Fixture

### `seed/test-users.ts`

- Connects to local Supabase using the service role key (env vars injected by devcontainer `setup.sh`)
- Creates test user via `supabase.auth.admin.createUser()` with confirmed email
- Inserts matching `profile`, `organization`, and `organizationmember` rows (course creation requires org context)
- Idempotent — checks if user exists first, skips if already seeded
- Test credentials: `testuser@classroomio.test` / `TestPass123!`

### `fixtures/auth.fixture.ts`

- Extends Playwright's base `test` fixture
- Provides an `authenticatedPage` fixture that loads `storageState` from `.auth/user.json`
- Gives step definitions a pre-authenticated `Page` object

### Setup Project Flow (runs before tests)

1. Seed script inserts test user into local Supabase
2. Playwright navigates to `/login`, fills credentials, submits
3. Saves browser `storageState` to `.auth/user.json`
4. All subsequent test projects reuse that session file

## Developer Workflow

```bash
# From repo root:
pnpm --filter e2e seed         # Seed test user (once, or after Supabase reset)
pnpm --filter e2e test          # Runs bddgen + playwright test
pnpm --filter e2e test:ui       # Interactive Playwright UI mode
pnpm --filter e2e report        # Open HTML report on port 9323
```

## Out of Scope (YAGNI)

- No page object model — only 2 flows, step definitions call Playwright directly
- No parallel test sharding
- No multiple browsers — Chromium only
- No visual regression testing
- No API-level testing (Vitest's domain)
- No CI integration — local dev only for now
- No custom fixtures beyond auth

These can all be added later when the test suite grows.
