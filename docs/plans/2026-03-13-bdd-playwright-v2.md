# BDD Tests with Gherkin & Playwright — Design Document

**Date:** 2026-03-13
**Status:** Draft
**Scope:** Initial setup — login and course creation flows only

---

## Goal

Establish BDD testing infrastructure so that critical user flows have automated regression coverage with human-readable Gherkin scenarios. Target maturity: **MVP** — get the framework working end-to-end with 2 flows, then expand.

**Success criteria:**
- Login and course creation scenarios pass green against the local devcontainer
- Playwright HTML report is accessible from the host machine
- A new developer can run the tests with no manual setup beyond `pnpm dev:container`

## Overview

Set up BDD testing for ClassroomIO using `playwright-bdd` (Gherkin integration on top of Playwright's native test runner). The initial scope covers two flows: login and course creation. Tests run against the local devcontainer environment and the Playwright HTML report is accessible from the host machine on port 9323.

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| BDD framework | `playwright-bdd` | Keeps Playwright's native runner, parallel execution, traces, and HTML report. Lighter than `@cucumber/cucumber`. |
| Package location | `packages/e2e/` | Standalone workspace package. Scales across apps, consistent with monorepo pattern. |
| Step definition org | Per feature file | Simple with only 2 flows. Refactor to domain grouping later if reuse grows. |
| Test data strategy | Supabase seed script | Reliable, flows run independently. No dependency on signup working. |
| Report port | 9323 | Playwright's default. Added to devcontainer port forwarding. |

### Package Name

The `package.json` `name` field must be `"e2e"` to match the `pnpm --filter e2e` commands in the developer workflow. The `pnpm-workspace.yaml` already includes `packages/*` so it will be auto-discovered.

### Dependencies

- `@playwright/test` — test runner
- `playwright-bdd` — Gherkin-to-Playwright integration
- `@supabase/supabase-js` — seed script (connects to local Supabase)

### Scripts (`packages/e2e/package.json`)

| Script | Command | Description |
|--------|---------|-------------|
| `generate` | `npx bddgen` | Generate Playwright specs from `.feature` files |
| `test` | `npx bddgen && npx playwright test` | Generate + run all BDD tests |
| `test:ui` | `npx bddgen && npx playwright test --ui-port=9324 --ui-host=0.0.0.0` | Interactive Playwright UI mode (web-based, accessible from host) |
| `report` | `npx playwright show-report --port 9323` | Serve HTML report on port 9323 |
| `seed` | `npx tsx seed/test-users.ts` | Insert test data into local Supabase (also runs automatically via `globalSetup`) |

## Playwright & playwright-bdd Configuration

### `playwright.config.ts`

- **Base URL:** `http://localhost:5173` (dashboard dev server)
- **Browser:** Chromium only
- **Test directory:** `.features-gen/` (generated by `bddgen`)
- **Reporter:** HTML reporter (port 9323)
- **Trace:** `on-first-retry`
- **`globalSetup`** — runs `seed/test-users.ts` automatically before all tests (idempotent, no manual seed step needed)
- **`defineBddConfig()`** — points `features` to `./features/`, `steps` to `./steps/`, and sets `importTestFrom: './fixtures/auth.fixture.ts'` so generated specs use custom fixtures
- **Locator strategy note:** The dashboard's `TextField` component wraps inputs in `<label>` but uses `<p>` for the label text — `getByLabel()` may not work reliably. Step definitions should prefer `getByPlaceholder()` or scoped CSS selectors as fallback. Verify locators against actual markup during implementation.

### Test Projects

Two Playwright projects (not monorepo packages):

1. **`setup`** — `testDir: './setup'`, `testMatch: '**/*.setup.ts'`. Runs `auth.setup.ts` which seeds test data, logs in via the UI, and saves `storageState` to `.auth/user.json`. Does NOT use `defineBddConfig` — this is a plain Playwright project.
2. **`tests`** — `testDir` set to `.features-gen/` (from `defineBddConfig`). Depends on `setup`. Runs all `.feature` flows with the saved auth state via `use: { storageState: '.auth/user.json' }`.

This means login feature tests exercise the actual login UI, while course creation tests skip login by reusing the stored session.

## Feature Files

### `features/login.feature`

```gherkin
Feature: Login

  Scenario: Successful login with valid credentials
    Given I am on the login page
    When I enter the test user credentials
    And I click the "Log In" button
    Then I should be redirected to the org courses page

  Scenario: Login fails with invalid password
    Given I am on the login page
    When I enter the test user email with password "wrongpassword"
    And I click the "Log In" button
    Then I should see an error message
```

### `features/course-creation.feature`

```gherkin
Feature: Course Creation

  Scenario: Create a live class course
    Given I am logged in
    And I am on the org courses page
    When I click the "Create Course" button
    And I select "Live Class" as the course type
    And I click the "Next" button
    And I enter a unique course title
    And I enter "A test course description" as the description
    And I click the "Finish" button
    Then I should see the created course in the course list

  Scenario: Create a self-paced course
    Given I am logged in
    And I am on the org courses page
    When I click the "Create Course" button
    And I select "Self Paced" as the course type
    And I click the "Next" button
    And I enter a unique course title
    And I enter "Self paced description" as the description
    And I click the "Finish" button
    Then I should see the created course in the course list
```

## Seed Script & Auth Fixture

### `seed/test-users.ts`

- Connects to local Supabase using the service role key
- **Env vars:** `setup.sh` must be updated to also inject `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` into `packages/e2e/.env`. Alternatively, the seed script can read from `apps/dashboard/.env` (where `PUBLIC_SUPABASE_URL` and `PRIVATE_SUPABASE_SERVICE_ROLE` already exist)
- Creates test user via `supabase.auth.admin.createUser()` with `email_confirm: true` (explicitly confirms email in `auth.users`)
- Inserts matching rows in this order:
  1. `profile` — linked to auth user id
  2. `organization` — with a known `siteName` (e.g., `testorg`) for URL assertions
  3. `organizationmember` — with `role_id = 1` (Admin), required for course creation through RLS (`is_user_in_group_with_role()`)
- Includes a localhost guard: refuses to run if `SUPABASE_URL` does not contain `localhost` or `127.0.0.1`
- Idempotent — checks if user exists first, skips if already seeded
- Test credentials read from env vars `TEST_USER_EMAIL` and `TEST_USER_PASSWORD` (defaults: `testuser@classroomio.test` / `TestPass123!`). Step definitions also read from the same env vars — credentials never appear in `.feature` files.

### `fixtures/auth.fixture.ts`

- Extends Playwright's base `test` fixture
- Provides an `authenticatedPage` fixture that loads `storageState` from `.auth/user.json`
- Gives step definitions a pre-authenticated `Page` object

### Prerequisites

- `supabase start` must have completed first — the BDD seed depends on `role` table rows (1=Admin, 2=Tutor, 3=Student) populated by `supabase/seed.sql`
- Dashboard dev server must be running (`pnpm dev:container`)

### Setup Project Flow (runs before tests)

1. `globalSetup` runs seed script automatically (idempotent — safe to re-run)
2. Playwright navigates to `/login`, fills credentials, submits
3. Saves browser `storageState` to `.auth/user.json`
4. All subsequent test projects reuse that session file

## Devcontainer Changes

1. Add ports `9323` and `9324` to `forwardPorts` in `.devcontainer/devcontainer.json` with labels "Playwright Report" and "Playwright UI".
2. Add Playwright browser installation to `.devcontainer/setup.sh`:
   ```bash
   cd /workspaces/classroomio/packages/e2e && npx playwright install --with-deps chromium
   ```
   This installs the Chromium binary and required OS libraries (libgbm, libnss3, etc.). Runs once on container creation.

## Follow-up

- **Remove Cypress:** Once Playwright BDD covers login and course creation, remove `cypress/`, `cypress.config.js`, the Cypress devDependency, and the root `"ci": "cypress run"` script. The existing 3 Cypress auth tests overlap with the new login feature.

## Out of Scope (YAGNI)

- No page object model — only 2 flows, step definitions call Playwright directly
- No parallel test sharding
- No multiple browsers — Chromium only
- No visual regression testing
- No API-level testing (Vitest's domain)
- No CI integration — local dev only for now
- No custom fixtures beyond auth

These can all be added later when the test suite grows.
